## 밤 작업

**요청:** 미리보기 화면 안의 인쇄 버튼(window.print())이 동작하지 않는 것 같다는 제보, 발견되는 필요한 수정사항은 모두 처리
**작업 내용:**

- Playwright로 iframe 안에서 (1) `window.print()` 직접 호출 (2) `window.open()`으로 새 창을 띄운 뒤 그 창에서 인쇄하는 패턴 두 가지를 재현 테스트: 두 경우 모두 beforeprint/afterprint 이벤트가 정상 발생하고 예외 없이 동작하는 것은 확인했으나, 새 창 인쇄 패턴이 더 안전하게 동작하도록 `allow-popups-to-escape-sandbox`를 sandbox 속성에 추가.
- 이전 body 속성 유실 수정과 함께 재검증: 두 수정 모두 정상 동작.

**결과:** 완료. `index.html`의 iframe sandbox에 `allow-popups-to-escape-sandbox` 추가. `node --check` 통과, Playwright로 회귀 테스트(중앙 정렬 + 인쇄 플로우) 통과.

## 심야 작업 (재조사)

**요청:** 사용자가 실제 급식컵 원본 코드 전체를 붙여넣고, 이전에 고쳤다고 보고한 중앙 정렬/하단 네비 문제가 여전히 재현된다고 제보
**작업 내용:**
- 실제 코드로 재검증한 결과, 저녁 작업에서 고친 body 태그 class 유실 버그는 이 파일에는 해당 없음(이 파일 `<body>`는 애초에 class가 없음)을 확인.
- **중앙 정렬 문제**: Playwright로 (1) CodeCanvas 없이 순수 브라우저에 동일 구조(`<div class="w-full max-w-md ... min-h-screen ...">`, `mx-auto` 없음)를 직접 열어본 결과와 (2) CodeCanvas 미리보기 iframe 결과가 완전히 동일(`left: 0`, 왼쪽 붙음)함을 실측 확인. → **CodeCanvas 버그 아님.** 원본 코드의 모바일 카드 div 자체에 `mx-auto`/부모 `flex justify-center`가 없어서 어떤 브라우저에서 열어도 왼쪽 정렬됨. 사용자에게 해당 div에 `mx-auto` 추가를 안내(사용자 프로젝트 코드라 직접 수정은 불가, 코드 스니펫만 전달).
- **공유 링크 하단 네비 문제**: `css/layout.css`의 `body.view-mode` 규칙(.main-content/.preview-section/.preview-container)이 전부 `100vh`만 사용 중임을 확인. 모바일 브라우저는 주소창 표시/숨김에 따라 `100vh`가 실제 보이는 영역보다 커지는 문제가 있어(공유 링크를 모바일 실기기로 열 때만 재현되고 데스크톱 에디터 미리보기/전체보기에서는 재현 안 되는 사용자의 증상과 정확히 일치), `sticky bottom-0` 네비가 화면 아래로 밀려 보이지 않는 원인으로 추정.
- 사용자 승인 후 `css/layout.css`의 세 규칙에 `100dvh`(동적 뷰포트 높이) 값을 `100vh` 다음 줄에 추가(구형 브라우저는 `100vh`로 자동 폴백, 모던 브라우저는 `100dvh`가 실제 보이는 영역 기준으로 적용). Playwright로 `CSS.supports('height','100dvh')` 및 view-mode 클래스 강제 적용 후 세 요소의 computed height가 `window.innerHeight`와 정확히 일치함을 확인, 회귀 없음.

**결과:** 완료. `css/layout.css` 3곳에 `100dvh` 폴백 추가. Playwright로 computed style 검증 통과. (모바일 실기기 주소창 동작 자체는 헤드리스 브라우저로 100% 재현 불가하여, 실기기에서 최종 확인은 사용자 몫으로 안내 필요)

## 새벽 작업 (진짜 원인 발견)

**요청:** 사용자가 실제 공유 링크(`https://code-canvas-fawn.vercel.app/?p=ea43719a-3ada-4af1-9356-30f337f550f5`)를 데스크톱 크롬 시크릿 창으로 열어도 하단 네비가 안 보인다고 스크린샷 2장과 함께 재차 제보. 이전 `100dvh` 수정은 실효가 없었음.
**작업 내용:**
- 데스크톱에서도 재현된다는 점에서 `100vh`/`100dvh` 모바일 주소창 가설은 기각. Playwright로 실제 라이브 공유 링크에 직접 접속해 DOM/레이아웃을 정밀 실측.
- `nav`의 `getBoundingClientRect()`, `getComputedStyle()`, `elementFromPoint()`는 전부 정상(위치·불투명도·표시 여부 이상 없음)인데 스크린샷에는 안 보이는 모순 확인. `nav.scrollIntoView()`를 호출하면 네비가 갑자기 보이는 현상을 발견하고 처음엔 "sticky 요소 초기 페인트 버그"로 오인.
- 하지만 리플로우만 강제(줌 토글)해서는 고쳐지지 않고, 진짜 스크롤이 있어야만 고쳐지는 것을 확인 → `scrollIntoView()`가 실제로 무엇을 움직이는지 좌표로 추적한 결과, **진짜 원인**을 특정:
  - `.app-container`는 `height: calc(100vh - var(--header-height))`(헤더 50px 제외)에 `overflow:hidden`. 뷰모드에서 헤더는 `display:none`이 되지만 이 높이 계산식 자체는 그대로 남아있었음.
  - 그런데 그 자식인 `.main-content`는 뷰모드 규칙으로 `height:100vh/100dvh`로 강제됨 → 부모(850px)보다 자식(900px)이 50px 더 커짐 → 부모의 `overflow:hidden`에 의해 하단 50px(정확히 하단 네비 영역)이 잘려서 안 보였던 것. sticky 페인트 버그가 아니라 **부모-자식 높이 불일치로 인한 클리핑**이었음.
  - (`scrollIntoView()`가 "고쳐 보인" 이유: `overflow:hidden` 컨테이너도 스크롤바만 없을 뿐 `scrollTop`은 프로그래밍적으로 움직일 수 있어서, scrollIntoView가 그 안쪽 컨테이너들의 scrollTop을 강제로 밀어 넣은 것 — 사용자 실기기에서는 스크롤 없이는 재현 안 될 문제)
- 라이브 페이지에 `body.view-mode .app-container { height:100vh; height:100dvh; }` 스타일을 임시 주입해 가설을 먼저 검증(스크롤 없이 바로 네비 노출 확인) 후, `css/layout.css`에 동일 규칙을 정식 추가.
- 로컬 dev 서버(`localhost:5183`)에서 실제 소스 파일 수정본으로 동일한 공유 링크(Supabase의 실제 프로젝트 데이터)를 새로고침만으로 재확인 — 스크롤/트릭 없이 하단 네비 4개 탭이 즉시 정상 노출됨을 스크린샷으로 확인.

**결과:** 완료. `css/layout.css`에 `body.view-mode .app-container` 높이 규칙 추가(100vh/100dvh 폴백 패턴 동일 적용). 라이브 URL 스타일 주입 검증 + 로컬 dev 서버 실제 파일 검증 두 단계 모두 통과. 이전 `100dvh` 수정(저녁 작업)은 결과적으로 무해하지만 실제 원인이 아니었음 — 진짜 원인은 이번에 고친 `.app-container` 높이 불일치. 배포(Vercel)까지 반영되려면 커밋 후 푸시 필요.

## 새벽 작업 (헤더 겹침 버그)

**요청:** 화면 폭이 좁아지면 상단 헤더의 프로젝트 이름 입력창 부분이 "저장됨" 상태 텍스트와 겹쳐서 깨져 보인다는 스크린샷 제보.
**작업 내용:**
- Playwright로 뷰포트를 768~1024px(태블릿 구간, 헤더가 한 줄 유지) 사이로 좁혀가며 재현. 780px에서 `.project-title-input`의 실제 렌더링 박스(width 163px)가 자신의 부모 `.header-left`(width 220px)보다 더 넓게 그려지며 옆의 `.header-center`("저장됨") 박스 위를 그대로 덮는 것을 좌표로 확인.
- 원인은 flexbox의 잘 알려진 함정: `<input>`은 기본적으로 `min-width: auto`라서 내용 크기 이하로 줄어들지 않고 부모 박스를 뚫고 넘쳐버림(`min-width:0`이 없었음).
- `min-width:0`만 주면 겹침은 사라지지만 입력창이 거의 안 보일 정도로 작아짐 → 추가로 공간을 확보하기 위해 점검한 결과, `.header-right`의 로그인 버튼이 `auth.js`의 `updateUI()`에서 `<span class="icon">/<span class="text">` 구조 없이 통짜 텍스트로 다시 그려지고 있어서, 기존의 "1024px 이하에서 버튼 텍스트 숨김" 반응형 규칙이 이 버튼에는 전혀 적용되지 않고 있었음(항상 "로그인" 텍스트가 표시되어 공간을 추가로 잡아먹음).
- 3곳 수정: (1) `css/layout.css` `.project-title-input`에 `min-width:0` 추가, (2) `css/responsive.css` 1024px 구간에 탐색기 버튼(`.header-left .btn .text`)도 텍스트 숨김 대상에 포함 + `.project-title-input`에 `min-width:50px`로 최소 가독성 확보, (3) `js/auth.js`의 로그인 버튼 렌더링을 다른 버튼들과 동일한 `icon`/`text` span 구조로 통일해 반응형 텍스트 숨김이 정상 적용되도록 수정.
- Playwright로 700 / 780 / 1100px 세 구간 모두 재검증: 780px에서 더 이상 겹침 없이 프로젝트명이 읽히고, 1100px(비대상 구간)에서는 기존처럼 모든 버튼 텍스트가 정상 노출되어 회귀 없음을 스크린샷으로 확인.

**결과:** 완료. `css/layout.css`, `css/responsive.css`, `js/auth.js` 3개 파일 수정. Playwright로 좁은 화면 겹침 해소 + 넓은 화면 회귀 없음 확인.
